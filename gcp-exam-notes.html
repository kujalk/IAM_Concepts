Google Cloud Security — Mind Map
Complete topic overview at a glance
Core Principles to Remember
🛡 Deny by Default
In GCP IAM, no binding = no access. Every request must have an explicit Allow binding. There is no implicit permission.
⬇ Inheritance flows DOWN
Org → Folder → Project → Resource. A binding at a higher level automatically applies to everything below it.
➕ Permissions are Additive
Multiple bindings on the same resource add together. Most-permissive wins — except explicit Deny Policies which override Allow.

Cloud Identity vs GCP IAM
Two different problems — Identity Provider vs Authorization Engine
🪪 Google Cloud Identity
Identity Provider (IdP) — Authentication = "Who are you?"
  • Manages users & groups
  • Handles login / authentication
  • SSO to GCP + external apps (SAML 2.0)
  • Enforces MFA policies
  • Device management
  • User lifecycle management (SCIM)
→
🔑 GCP IAM
Authorization Engine — "What can you do?"
  • Manages role bindings
  • Controls resource access
  • Enforces least privilege
  • Evaluates policies at runtime
  • Logs all access decisions
  • Works with ANY Google identity
Authentication + Authorization Flow
1
👤 User attempts to access a GCP resource
e.g., opens Cloud Console or calls gcloud / API
2
🪪 Cloud Identity (or external IdP) authenticates the user
Validates credentials, enforces MFA, issues identity token
3
🎟 Google issues an identity token (JWT)
Token contains email / sub claims identifying the user
4
🔑 GCP IAM evaluates token against the resource's policy bindings
Checks if the identity's email/sub matches any Allow binding for the requested role
5
✅ Access granted or ❌ denied based on role bindings
IAM doesn't care how you authenticated — only that your identity matches a binding

3
Workload Identity Federation (WIF)
Keyless authentication for external workloads — no JSON key files
❌ Service Account Key Files
  • JSON keys are long-lived (no expiry)
  • Must be securely distributed to workloads
  • Stored in CI/CD secrets, env vars, k8s secrets
  • If leaked → full SA access until manually revoked
  • Hard to rotate without breaking deployments
vs
✓ Workload Identity Federation
  • Keyless — zero JSON key files
  • Short-lived tokens (max 1 hour)
  • Tokens scoped to specific workload identity
  • Attribute conditions restrict to repo/branch/env
  • Automatic rotation — each job gets a fresh token
Token Exchange Flow — 6 Steps
1
⚙ External workload starts
GitHub Actions runner / AWS Lambda / GKE Pod begins execution in its native environment
2
🎟 Gets external identity token
GitHub → OIDC JWT  |  AWS → STS GetCallerIdentity  |  GKE → K8s SA JWT  |  Azure → OIDC JWT from Azure AD
3
🔄 Exchange token at GCP STS (sts.googleapis.com/v1/token)
Verifies token signature (JWKS URI) · Checks issuer URI · Checks audience · Evaluates attribute conditions
4
🎫 Receives short-lived federated token (max 1 hour)
Principal: principalSet://iam.googleapis.com/projects/.../workloadIdentityPools/...
5
🎭 Impersonates a Service Account (optional)
Requires roles/iam.workloadIdentityUser granted on the SA to the federated principal
6
✅ Calls GCP APIs as Service Account
Short-lived SA access token. No JSON key file was used at ANY point. Entire chain is ephemeral.
⚠
Issuer URI (iss) — Identifies WHO issued the external token. Configured in the WIF Pool Provider. GCP uses this to find the JWKS signing keys.
Audience (aud) — Identifies WHO the token is intended for. Must match what you configure in the WIF Provider's allowed audiences.
Typical audience: //iam.googleapis.com/projects/PROJECT_NUM/locations/global/workloadIdentityPools/POOL_ID/providers/PROVIDER_ID

Workforce Identity Pools are an org-level construct (unlike Workload Identity Pools which are project-level). A single pool can be shared across all projects in your organization.
Key Concepts
Workforce Identity Pool
Org-level container for external human identities. Created at Organization level, shared across projects.

gcloud iam workforce-pools create my-corp-pool --organization=ORG_ID --location=global
Pool Provider
Configures the external IdP. Supports OIDC and SAML 2.0. Defines attribute mapping and conditions.

--issuer-uri=https://my-company.okta.com
Attribute Mapping
Maps IdP token claims → Google attributes. google.subject is required. google.email enables email-based IAM bindings.

google.subject = assertion.sub
attribute.department = assertion.dept
IAM Binding with WIF Principal
Bind roles to external users via workforce pool identity. No Google account needed.

--member="principalSet://iam.googleapis.com/locations/global/workforcePools/POOL/attribute.dept/eng"

💡 Why Pub/Sub?
Decouples log producer (GCP) from consumer (SIEM). SIEM can be offline; messages queue up to 7 days. Supports multiple subscribers.
🔒 Log Types to Sink
Admin Activity Logs — always on
Data Access Logs — opt-in, high volume
System Event Logs — always on
Policy Denied Logs — opt-in
⚠ IAM Gotcha
The Pub/Sub topic can be in a different project (the SIEM project). The sink SA needs pubsub.publisher on that cross-project topic.

GCP Encryption Guide
At-rest, in-transit, and in-use — GMEK / CMEK / CSEK / EKM / Confidential Computing
Envelope Encryption — How GCP Always Works
DEK = Data Encryption Key  |  KEK = Key Encryption Key  |  The KEK never leaves KMS; only the wrapped DEK is stored alongside the data
Encryption Key Types
🔒 GMEK — Google-Managed Encryption Keys
Default   Zero config

How: Google owns, rotates, and manages entire key hierarchy. AES-256 per data chunk automatically.
Best for: Dev/test, non-regulated workloads
Cons: No customer control, no audit trail, no crypto-shredding
🔑 CMEK — Customer-Managed Encryption Keys
Recommended for regulated

How: You create Key Ring + CryptoKey in Cloud KMS. Service agent gets Encrypter/Decrypter role. All ops logged in Cloud Audit Logs.
Best for: HIPAA, PCI-DSS, SOC2, data sovereignty
Key feature: Crypto-shredding — destroy key → data permanently gone
🗝️ CSEK — Customer-Supplied Encryption Keys
Highest operational burden

How: You supply raw AES-256 key on EVERY API call. Google uses it in-memory, never stores it.
Services: Cloud Storage (object-level) + Compute Engine disks only
⚠ Warning: Lose the key = permanent irrecoverable data loss. CLI/API only (no Console).
🏛️ EKM — External Key Manager
Ultimate separation

How: Keys live in 3rd-party KMS (Thales, Fortanix) outside GCP entirely. Cloud KMS references external key via URI.
Feature: Key Access Justifications — see WHY Google requests decryption. Kill switch — revoke instantly.
Best for: Schrems II, GDPR sovereign data, government/defense
🛡️ Confidential Computing — Encryption In-Use
How: Hardware TEE (AMD SEV / Intel TDX) encrypts VM memory. Hypervisor and host OS cannot read VM memory.
EKM UDE: External KMS wraps DEKs so ONLY attested Confidential VMs can unwrap them.
Best for: Multi-party computation, sensitive ML/AI, zero-trust architectures

Encryption in Transit + Quick Reference
Data in motion + key facts for the exam
Encryption in Transit
🌐 TLS/HTTPS
All GCP APIs enforce TLS 1.2+ by default. Data between your browser/app and Google Front End (GFE) is always encrypted. You cannot turn this off.
🔄 ALTS
Application Layer Transport Security — Google's internal mutual-auth protocol for service-to-service communication within Google data centers. Used by internal Google services.
🔗 VPN / Interconnect
Cloud VPN (IPsec) encrypts data between on-prem and GCP. Dedicated/Partner Interconnect provides private connectivity (not internet). Use Cloud VPN over Interconnect for additional encryption.
🕸️ mTLS & Service Mesh
Istio/Anthos Service Mesh enables mutual TLS between microservices in GKE for zero-trust networking. Every pod-to-pod call is authenticated and encrypted.
🎯 Exam Quick Reference — Key Facts
IAM Fundamentals
• Deny by default — no binding = no access
• Policies inherit downward: Org → Folder → Project → Resource
• Permissions are additive; Deny Policies override Allow
• allUsers = unauthenticated public  |  allAuthenticatedUsers = any Google account (not just your org)
• Basic roles (viewer/editor/owner) = avoid in prod — too permissive
Workload vs Workforce Identity Federation
• Workload = machine identities (GitHub Actions, AWS, K8s pods) exchanging tokens for SA credentials — project-level pools
• Workforce = human identities (employees) from Okta/Azure AD accessing GCP — org-level pools
• Both use GCP STS token exchange; neither requires Google accounts
• Workload: roles/iam.workloadIdentityUser on SA granted to federated principal
• Workforce: bind roles to principalSet://iam.googleapis.com/locations/global/workforcePools/POOL/...
Load Balancing Decision Rules
• Internet clients + HTTP(S) + global → Global External Application LB (Premium Tier)
• Internet clients + TCP/UDP + global → Global External Proxy Network LB
• Need original client IP preserved → Regional External Passthrough NLB (passthrough mode)
• Microservices in same VPC → Regional Internal Application LB (L7) or Regional Internal Passthrough NLB (L4)
• Internal traffic spanning regions → Cross-Region Internal Application LB (Premium Tier, internal)
Encryption Key Selection Rules
• Default / no requirement → GMEK (free, zero-config, all services)
• Compliance (HIPAA/PCI/SOC2) + audit trail → CMEK via Cloud KMS
• Sole custodian of keys, 2 services only → CSEK (CLI/API only, key on every call)
• Keys must NEVER enter Google infrastructure → EKM (Thales, Fortanix)
• Schrems II / EU data sovereignty → EKM + Key Access Justifications
• Encrypt data in memory → Confidential VMs (AMD SEV / Intel TDX)
Cloud Logging → SIEM Critical Points
• Log Sink auto-creates a service account: p-NNNNN@gcp-sa-logging.iam.gserviceaccount.com
• That SA needs pubsub.publisher on the destination Pub/Sub topic (can be cross-project)
• Pub/Sub message retention: up to 7 days
• Admin Activity Logs + System Event Logs: always on, free
• Data Access Logs: must opt-in, high volume, can incur cost
